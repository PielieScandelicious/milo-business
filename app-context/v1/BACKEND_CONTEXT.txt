1. SYSTEM OVERVIEW
================================================================================

Primary Purpose:
----------------
Milo (internally "Scandalicious") is a receipt scanning and expense tracking REST API
backend that powers an iOS mobile application. The system enables users to:
- Photograph grocery receipts and extract item data via OCR
- Automatically categorize purchases using AI
- Assign health scores to food items
- Set and track monthly budgets with AI-powered insights
- Query spending history through an AI chat assistant (Dobby)
- View analytics and spending trends

High-Level Architecture:
------------------------
- Architecture Pattern: Modular monolithic REST API
- Request Flow: FastAPI routes -> Dependencies (auth/db) -> Services -> Repositories -> Database
- Processing Model: Async/await throughout using Python asyncio
- API Versioning: Dual API versions (v1 legacy, v2 recommended) at /api/v1 and /api/v2

Key Architectural Decisions:
----------------------------
1. Repository pattern separates data access from business logic
2. Service layer encapsulates all external API interactions (Veryfi, Gemini, Claude)
3. Dependency injection via FastAPI's Depends() for auth and database sessions
4. Custom exception hierarchy with global exception handlers for consistent error responses
5. Streaming responses manage their own database sessions to prevent pool exhaustion
6. AI insights are cached to reduce API costs and improve response times

Why Services Are Separated From Controllers (API Routes):
---------------------------------------------------------
- Routes handle HTTP concerns only: validation, authentication, response formatting
- Services encapsulate business logic: orchestration, external API calls, data transformation
- Repositories handle data access: SQL queries, ORM operations
- This separation enables: unit testing, service reuse, clear responsibility boundaries

4. DATA FLOW
================================================================================

STANDARD REQUEST FLOW (Example: POST /api/v2/receipts/upload)
------------------------------------------------------------------------------

Step 1: HTTP Entry (app/api/v2/receipts.py:upload_receipt)
    Client sends multipart form with image file
    FastAPI validates content type, parses UploadFile

Step 2: Dependency Resolution
    get_db() -> Creates AsyncSession from async_session_maker
    get_current_db_user() -> Validates Firebase token -> Returns User model

Step 3: Rate Limit Check (app/services/rate_limit_service.py)
    RateLimitService.check_receipt_rate_limit(firebase_uid)
    -> RateLimitRepository.get_or_create()
    -> Returns ReceiptRateLimitStatus with allowed flag

Step 4: Business Logic (app/services/receipt_processor_v2.py:process_receipt)
    a) ImageValidator.validate_content_type() - Check supported formats
    b) ReceiptRepository.create() - Create receipt record (status=PROCESSING)
    c) ImageValidator.raise_if_invalid() - Validate image quality
    d) VeryfiService.extract_receipt_data() - Call Veryfi OCR API
       - Base64 encode image
       - POST to https://api.veryfi.com/api/v8/partner/documents
       - Parse response into VeryfiExtractionResult
    e) Check duplicate flag - If duplicate, delete receipt and return early
    f) CategorizationServiceGemini.categorize_items() - Call Gemini API
       - Format items as numbered list
       - Send to gemini-2.0-flash with system prompt
       - Parse JSON response into CategorizedItem list
    g) TransactionRepository.create() - Create transaction for each item
    h) ReceiptRepository.update() - Update receipt (status=COMPLETED)

Step 5: Response
    Return ReceiptUploadResponse with:
    - receipt_id, status, store_name, receipt_date, total_amount
    - items_count, transactions[], warnings[], is_duplicate

Step 6: Session Commit
    get_db() context manager commits transaction
    If exception: automatic rollback

STREAMING CHAT FLOW (Example: POST /api/v2/chat/stream)
------------------------------------------------------------------------------

Step 1: Initial validation and rate limit check in route handler
Step 2: Return StreamingResponse immediately with stream_response generator
Step 3: stream_response() creates its own async session (prevents pool exhaustion)
Step 4: DobbyChatServiceGemini.chat_stream():
    a) Fetch UserProfile from database
    b) Build transaction context (last 365 days, aggregated statistics)
    c) Call Gemini API with streaming: client.models.generate_content_stream()
    d) Yield SSE events: {"type": "text", "content": "..."}
Step 5: On completion: increment rate limit, commit session, yield done event
Step 6: Session closes when generator exhausts

BUDGET AI SUGGESTION FLOW (Example: GET /api/v2/budgets/ai-suggestion)
------------------------------------------------------------------------------

Step 1: Auth & validation (get months parameter, default 3, max 12)
Step 2: Check for cached valid suggestion (< 24 hours old, matching months)
Step 3: If cached: return immediately from database
Step 4: If not cached:
    a) Gather spending data (BudgetAIService._gather_spending_data)
       - Category totals, trends, recent transactions
    b) Build AI prompt (context + system instruction)
    c) Call Gemini API (gemini-2.0-flash)
    d) Parse JSON response into structured AIBudgetSuggestionResponse
    e) Store insight in database (BudgetAIInsightRepository.create)
    f) Invalidate old suggestions
Step 5: Return AIBudgetSuggestionResponse

5. AUTHENTICATION & AUTHORIZATION
================================================================================

Authentication Flow:
--------------------
1. iOS app authenticates user via Firebase (email/password, Google, Apple)
2. App obtains Firebase ID token (JWT)
3. App sends token in Authorization header: "Bearer <firebase_id_token>"
4. Backend verifies token with Firebase Admin SDK

Token Verification (app/core/security.py):
------------------------------------------
    auth.verify_id_token(token) -> decoded_token
    Returns: {uid, email, name, ...}

User Resolution (app/api/deps.py):
----------------------------------
    get_current_user() -> FirebaseUser (uid, email, name)
    get_current_db_user() -> User (SQLAlchemy model)
        - Looks up user by firebase_uid
        - Auto-creates user on first authentication

Key Classes:
------------
    FirebaseUser (security.py:48-54)
        Lightweight dataclass with uid, email, name

    User (models/user.py:17-45)
        SQLAlchemy model with:
        - id (UUID, primary key)
        - firebase_uid (unique, indexed)
        - email, display_name, is_active
        - Relationships: transactions, receipts, profile, budget, ai_insights

Authorization Model:
--------------------
- All data is user-scoped via user_id foreign keys
- Repository methods filter by user_id: get_by_id_and_user()
- No role-based access control - single user type
- Rate limits are per-user via firebase_uid
- One budget per user (unique constraint)

Rate Limiting:
--------------
- Messages: 100 per 30-day rolling window (RATE_LIMIT_MESSAGES)
- Receipts: 15 per 30-day rolling window (RATE_LIMIT_RECEIPTS)
- Enforced in route handlers before processing
- Counter incremented only on successful operation (via callback pattern)

7. BUDGET FEATURE (COMPLETE DOCUMENTATION)
================================================================================

OVERVIEW
--------
The budget feature enables users to set monthly spending goals and receive
AI-powered insights about their spending habits. It includes:
- Manual budget creation with optional category allocations
- Real-time progress tracking against budget
- AI-generated budget suggestions based on spending history
- Daily AI check-ins with personalized tips
- Instant AI feedback on new receipts
- Monthly AI reports with grades and recommendations

CORE BUDGET MANAGEMENT
----------------------

Budget Model Fields:
- monthly_amount: Total monthly budget in EUR (required)
- category_allocations: Optional per-category budgets (JSONB)
- notifications_enabled: Whether to send budget alerts (default: true)
- alert_thresholds: Percentage triggers for alerts (default: [50, 75, 90])

Category Allocation Strategies:
1. Explicit Allocations: User defines specific amounts per category
2. Auto-Distribution: System calculates based on historical spending percentages

Budget Progress Calculation:
- Total spent this month vs. monthly_amount
- Days remaining in month
- Daily budget remaining = (monthly_amount - total_spent) / days_remaining
- Category-level progress (if allocations exist)

BUDGET SERVICE (app/services/budget_service.py)
-----------------------------------------------

Methods:
- get_current_month_spend(user_id) -> float
    Returns total spending for current calendar month

- get_current_month_spend_by_category(user_id) -> Dict[str, float]
    Returns category -> amount mapping for current month

- get_historical_category_percentages(user_id, months) -> Dict[str, float]
    Calculates average percentage per category over N months
    Used for auto-distributing budget across categories

- get_budget_progress(user_id) -> BudgetProgressResponse
    Full progress report with:
    - total_budget, total_spent, remaining
    - percentage_used, daily_budget_remaining
    - days_remaining, days_elapsed
    - categories: List[CategoryProgress] (if allocations exist)
    - projected_end_of_month (based on current rate)

- get_budget_suggestion(user_id, months=3) -> BudgetSuggestionResponse
    Non-AI budget recommendation based on:
    - Average monthly spending over N months
    - Rounded to nearest 5 EUR
    - Includes savings options (10%, 20% reductions)
    - Category breakdown with percentages

8. RECEIPT PROCESSING PIPELINE
================================================================================

UPLOAD FLOW (app/services/receipt_processor_v2.py)
--------------------------------------------------

1. Validation Phase:
   - Content type check (image/jpeg, image/png, application/pdf)
   - File size check (max 20MB)
   - Image quality check (min 300x400px)
   - PDF conversion if needed

2. OCR Phase (Veryfi API):
   - Base64 encode image
   - POST to Veryfi API v8
   - Extract: vendor_name, date, total, line_items
   - Check duplicate flag

3. Duplicate Detection:
   - Veryfi provides duplicate score and reference
   - If duplicate: delete receipt record, return is_duplicate: true
   - Prevents double-counting same receipt

4. Categorization Phase (Gemini):
   - Format items as numbered list
   - Send to Gemini 2.0 Flash
   - For each item: category (1 of 17), health_score (0-5)
   - Parse JSON response

5. Storage Phase:
   - Create Transaction for each line item
   - Update Receipt with totals and status
   - Commit to database

6. Response:
   - Return ReceiptUploadResponse
   - Include all transactions
   - Include any warnings

RECEIPT STATUSES
----------------
- pending: Created but not processed
- processing: Currently being processed
- completed: Successfully processed
- failed: Processing error (error_message populated)

11. RATE LIMITING SYSTEM
================================================================================

IMPLEMENTATION (app/services/rate_limit_service.py)
---------------------------------------------------

Two-Tier System:

Chat Messages:
- Limit: 100 per 30-day period
- Tracked per firebase_uid
- Counter in user_rate_limits table

Receipt Uploads:
- Limit: 15 per 30-day period
- Tracked per firebase_uid
- Counter in user_rate_limits table

Period Management:
- Period starts on first usage
- Period ends 30 days later
- Auto-resets when period expires
- New period starts on next usage

Callback Pattern:
- Check returns RateLimitStatus with allowed flag
- Status includes increment_callback
- Callback called only after successful operation
- Prevents counting failed attempts

Status Endpoints:
- GET /rate-limit - Chat message status
- GET /rate-limit/receipts-status - Receipt upload status

Response Headers (on limited endpoints):
- X-RateLimit-Limit: Maximum allowed
- X-RateLimit-Remaining: Remaining in period
- X-RateLimit-Reset: Unix timestamp when reset
