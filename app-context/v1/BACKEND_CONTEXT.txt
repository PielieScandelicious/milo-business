1. SYSTEM OVERVIEW
================================================================================

Primary Purpose:
Scandalicious is a receipt scanning and expense tracking REST API backend that
powers an iOS mobile application. The system enables users to photograph grocery
receipts, automatically extract item data via OCR, categorize purchases using AI,
assign health scores to food items, and query their spending history through an
AI-powered chat assistant.

High-Level Architecture:
- Architecture Pattern: Modular monolithic REST API
- Request Flow: FastAPI routes -> Dependencies (auth/db) -> Services -> Repositories -> Database
- Processing Model: Async/await throughout using Python asyncio
- API Versioning: Dual API versions (v1 legacy, v2 recommended) at /api/v1 and /api/v2

Key Architectural Decisions:
1. Repository pattern separates data access from business logic
2. Service layer encapsulates all external API interactions (Veryfi, Gemini, Claude)
3. Dependency injection via FastAPI's Depends() for auth and database sessions
4. Custom exception hierarchy with global exception handlers for consistent error responses
5. Streaming responses manage their own database sessions to prevent pool exhaustion

Why Services Are Separated From Controllers (API Routes):
- Routes handle HTTP concerns only: validation, authentication, response formatting
- Services encapsulate business logic: orchestration, external API calls, data transformation
- Repositories handle data access: SQL queries, ORM operations
- This separation enables: unit testing, service reuse, clear responsibility boundaries

4. DATA FLOW
================================================================================

STANDARD REQUEST FLOW (Example: POST /api/v2/receipts/upload)
------------------------------------------------------------------------------

Step 1: HTTP Entry (app/api/v2/receipts.py:upload_receipt)
    Client sends multipart form with image file
    FastAPI validates content type, parses UploadFile

Step 2: Dependency Resolution
    get_db() -> Creates AsyncSession from async_session_maker
    get_current_db_user() -> Validates Firebase token -> Returns User model

Step 3: Rate Limit Check (app/services/rate_limit_service.py)
    RateLimitService.check_receipt_rate_limit(firebase_uid)
    -> RateLimitRepository.get_or_create()
    -> Returns ReceiptRateLimitStatus with allowed flag

Step 4: Business Logic (app/services/receipt_processor_v2.py:process_receipt)
    a) ImageValidator.validate_content_type() - Check supported formats
    b) ReceiptRepository.create() - Create receipt record (status=PROCESSING)
    c) ImageValidator.raise_if_invalid() - Validate image quality
    d) VeryfiService.extract_receipt_data() - Call Veryfi OCR API
       - Base64 encode image
       - POST to https://api.veryfi.com/api/v8/partner/documents
       - Parse response into VeryfiExtractionResult
    e) Check duplicate flag - If duplicate, delete receipt and return early
    f) CategorizationServiceGemini.categorize_items() - Call Gemini API
       - Format items as numbered list
       - Send to gemini-2.0-flash with system prompt
       - Parse JSON response into CategorizedItem list
    g) TransactionRepository.create() - Create transaction for each item
    h) ReceiptRepository.update() - Update receipt (status=COMPLETED)

Step 5: Response
    Return ReceiptUploadResponse with:
    - receipt_id, status, store_name, receipt_date, total_amount
    - items_count, transactions[], warnings[], is_duplicate

Step 6: Session Commit
    get_db() context manager commits transaction
    If exception: automatic rollback

STREAMING CHAT FLOW (Example: POST /api/v2/chat/stream)
------------------------------------------------------------------------------

Step 1: Initial validation and rate limit check in route handler
Step 2: Return StreamingResponse immediately with stream_response generator
Step 3: stream_response() creates its own async session (prevents pool exhaustion)
Step 4: DobbyChatServiceGemini.chat_stream():
    a) Fetch UserProfile from database
    b) Build transaction context (last 365 days, aggregated statistics)
    c) Call Gemini API with streaming: client.models.generate_content_stream()
    d) Yield SSE events: {"type": "text", "content": "..."}
Step 5: On completion: increment rate limit, commit session, yield done event
Step 6: Session closes when generator exhausts

5. AUTH & PERMISSIONS
================================================================================

Authentication Flow:
1. iOS app authenticates user via Firebase (email/password, Google, Apple)
2. App obtains Firebase ID token (JWT)
3. App sends token in Authorization header: "Bearer <firebase_id_token>"
4. Backend verifies token with Firebase Admin SDK

Token Verification (app/core/security.py):
    auth.verify_id_token(token) -> decoded_token
    Returns: {uid, email, name, ...}

User Resolution (app/api/deps.py):
    get_current_user() -> FirebaseUser (uid, email, name)
    get_current_db_user() -> User (SQLAlchemy model)
        - Looks up user by firebase_uid
        - Auto-creates user on first authentication

Key Classes:
    FirebaseUser (security.py:48-54)
        Lightweight dataclass with uid, email, name

    User (models/user.py:17-45)
        SQLAlchemy model with:
        - id (UUID, primary key)
        - firebase_uid (unique, indexed)
        - email, display_name, is_active
        - Relationships: transactions, receipts, profile

Authorization Model:
- All data is user-scoped via user_id foreign keys
- Repository methods filter by user_id: get_by_id_and_user()
- No role-based access control - single user type
- Rate limits are per-user via firebase_uid

Rate Limiting:
- Messages: 100 per 30-day rolling window (RATE_LIMIT_MESSAGES)
- Receipts: 15 per 30-day rolling window (RATE_LIMIT_RECEIPTS)
- Enforced in route handlers before processing
- Counter incremented only on successful operation (via callback pattern)

CUSTOM EXCEPTION HIERARCHY (app/core/exceptions.py)
------------------------------------------------------------------------------
ScandaliciousException (base class)
├── ReceiptProcessingError     -> 422 Unprocessable Entity
├── ImageValidationError       -> 400 Bad Request
├── ClaudeAPIError             -> 503 Service Unavailable
├── VeryfiAPIError             -> 503 Service Unavailable
├── GeminiAPIError             -> 503 Service Unavailable
├── ResourceNotFoundError      -> 404 Not Found
├── PermissionDeniedError      -> 403 Forbidden
└── RateLimitExceededError     -> 429 Too Many Requests

Each exception has: message (str), details (dict)
Global handlers in main.py map exceptions to JSON responses:
    {"error": "error_type", "message": "...", "details": {...}}

LOGGING CONFIGURATION (app/main.py:25-35)
------------------------------------------------------------------------------
- Root logger: DEBUG if settings.DEBUG else INFO
- Suppressed loggers (set to WARNING):
  - urllib3, cachecontrol, httpx, httpcore, google, firebase_admin
- Usage: logger = logging.getLogger(__name__)

SETTINGS PATTERN (app/config.py)
------------------------------------------------------------------------------
@lru_cache() caches Settings instance
Settings class uses pydantic-settings:
- Loads from environment variables
- Supports .env file
- Type coercion and validation

Key settings:
- DATABASE_URL, DATABASE_POOL_SIZE, DATABASE_MAX_OVERFLOW
- FIREBASE_SERVICE_ACCOUNT or GOOGLE_APPLICATION_CREDENTIALS
- ANTHROPIC_API_KEY, GEMINI_API_KEY
- VERYFI_CLIENT_ID, VERYFI_CLIENT_SECRET, VERYFI_USERNAME, VERYFI_API_KEY
- MAX_UPLOAD_SIZE_MB (20), ALLOWED_EXTENSIONS (pdf, jpg, jpeg, png)
- CORS_ORIGINS, DEBUG

DEPENDENCY INJECTION (app/api/deps.py)
------------------------------------------------------------------------------
get_db() -> AsyncGenerator[AsyncSession, None]
    Creates session from async_session_maker
    Auto-commits on success, rollbacks on exception

get_current_db_user() -> User
    Combines get_current_user() (Firebase) + UserRepository.get_by_firebase_uid()
    Auto-provisions user on first authentication

IMAGE VALIDATOR (app/services/image_validator.py)
------------------------------------------------------------------------------
ImageValidator class with validation rules:
- MIN_WIDTH: 300px, MIN_HEIGHT: 400px
- MAX_FILE_SIZE_MB: 20MB
- SUPPORTED_FORMATS: JPEG, PNG
- SUPPORTED_CONTENT_TYPES: image/jpeg, image/jpg, image/png, application/pdf

Methods:
- validate_content_type() - Quick content type check
- validate() - Full validation returning ValidationResult
- raise_if_invalid() - Validate and raise ImageValidationError if invalid

RATE LIMIT CALLBACK PATTERN (app/services/rate_limit_service.py)
------------------------------------------------------------------------------
RateLimitStatus dataclass includes:
    _increment_callback: Optional[Callable[[], Awaitable[None]]]

This enables:
1. Check rate limit before processing
2. Process request
3. Only increment counter on success: await rate_status.increment_on_success()

Callback creates its own session, safe for streaming responses.

7. EXTERNAL CONNECTIONS
================================================================================

VERYFI OCR API (app/services/veryfi_service.py)
------------------------------------------------------------------------------
Endpoint: https://api.veryfi.com/api/v8/partner/documents
Purpose: Extract structured data from receipt images
Method: POST
Authentication:
    Headers:
        CLIENT-ID: {client_id}
        AUTHORIZATION: apikey {username}:{api_key}
Payload:
    {
        "file_data": "<base64_encoded_image>",
        "file_name": "receipt.jpg",
        "auto_delete": true,
        "boost_mode": false
    }
Response: VeryfiExtractionResult dataclass:
    - vendor_name, date, total, subtotal, tax
    - line_items: List[VeryfiLineItem]
    - currency_code, ocr_text
    - is_duplicate, duplicate_score
Timeout: 60 seconds
Error handling: VeryfiAPIError with error_type (authentication, rate_limit, api_error, timeout)

GOOGLE GEMINI API (app/services/categorization_service_gemini.py)
------------------------------------------------------------------------------
Client: google.genai.Client(api_key=...)
Model: gemini-2.0-flash
Purpose: Categorize receipt items and assign health scores
Method: client.models.generate_content()
Config:
    GenerateContentConfig(
        system_instruction=SYSTEM_PROMPT,
        max_output_tokens=4096,
        temperature=0.1
    )
Input: Formatted item list with store name
Output: JSON with store_name and items[{index, item_name, category, health_score}]
Error handling: GeminiAPIError

GOOGLE GEMINI CHAT (app/services/dobby_chat_service_gemini.py)
------------------------------------------------------------------------------
Model: gemini-2.0-flash
Purpose: AI chat assistant for spending insights
Methods:
    - chat(): Non-streaming response
    - chat_stream(): AsyncGenerator yielding text chunks
Config:
    GenerateContentConfig(
        system_instruction=DOBBY_SYSTEM_PROMPT,
        max_output_tokens=4096,
        temperature=0.7
    )
Context provided to AI:
    - User profile (first_name, last_name, gender)
    - Transaction data (last 365 days)
    - Category totals, store totals, monthly spending
    - Recent transactions (last 100)
Error handling: GeminiAPIError

ANTHROPIC CLAUDE API (app/services/claude_service.py - V1 only)
------------------------------------------------------------------------------
Model: claude-sonnet-4-20250514
Purpose: V1 chat assistant
Client: anthropic.Client(api_key=...)
Method: client.messages.create()
Used by: DobbyChatService (V1)
Error handling: ClaudeAPIError

FIREBASE ADMIN SDK (app/core/security.py)
------------------------------------------------------------------------------
Initialization options:
    1. GOOGLE_APPLICATION_CREDENTIALS (file path)

Method: firebase_admin.auth.verify_id_token(token)
